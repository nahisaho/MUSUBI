# 製品コンテキスト

## 概要
このドキュメントは、ビジネスコンテキスト、製品目的、コア機能を定義します。AIエージェントが開発決定の「なぜ」を理解し、ビジネス目標との整合性を確保するのに役立ちます。

**最終更新日**: 2025-01-16 by @steering

## 製品ビジョン

### 構築しているもの
**MUSUBI (むすび - "結び" - "繋ぐ/結ぶ")** - 憲法ガバナンスとマルチエージェントサポートを通じて、仕様、設計、コードを結びつける究極の仕様駆動開発ツールです。

MUSUBIは包括的なCLI配布SDDフレームワークで、以下を提供します:
- 7つのAIコーディングエージェントサポート (Claude Code, GitHub Copilot, Cursor, Gemini CLI, Codex CLI, Qwen Code, Windsurf)
- Claude Code専用の25の専門スキル (Skills API専用)
- 品質ガバナンスのための9つの憲法条項を実施
- 自動更新されるプロジェクトメモリ (ステアリングシステム) を維持
- 明確な仕様のためのEARS要件フォーマットを使用
- グリーンフィールド (0→1) とブラウンフィールド (1→n) の両方のワークフローを可能に

### 解決している問題
AIコーディングアシスタントは開発速度を劇的に向上させましたが、**品質とトレーサビリティが犠牲になっています**:

**ペインポイント**:
- **曖昧な要件**: "ユーザー認証を追加してください" → どの標準? どの機能? どのセキュリティレベル?
- **失われたトレーサビリティ**: コードは存在するが、なぜ? どの要件を満たす? どのテストで検証?
- **一貫性のない品質**: 異なるAIエージェントが異なるコードスタイルとアーキテクチャを生成
- **ガバナンスなし**: アーキテクチャレビューや憲法準拠なしで機能が実装される
- **ブラウンフィールドの混乱**: 影響分析なしの既存コード変更がリグレッションを引き起こす
- **マルチエージェント非互換**: 1つのAIエージェント用に構築されたツールが他と動作しない

### ターゲットユーザー
**プライマリオーディエンス**: AIコーディングアシスタントを使用する開発者 (Claude Code, GitHub Copilot, Cursor等)
- **役割/肩書**: フルスタック開発者、バックエンドエンジニア、DevOpsエンジニア、テックリード
- **会社規模**: スタートアップから大企業 (AI支援開発を使用するあらゆるチーム)
- **技術レベル**: 中級から上級 (CLIツール、git、markdownに精通)
- **AI経験**: AIコーディングエージェントを使用中または評価中

**セカンダリオーディエンス**: エンジニアリングマネージャーとアーキテクト
- AI支援開発全体で品質と一貫性を確保する必要
- 要件からコードへのトレーサビリティが必要
- アーキテクチャ標準の実施が必要 (憲法ガバナンス)

## コア機能

### 必須機能 (実装済み)

1. **マルチエージェントサポート (7プラットフォーム)**
   - **ユーザー価値**: 好みのAIコーディングエージェントでMUSUBIを使用
   - **機能**: Claude Code, GitHub Copilot, Cursor, Gemini CLI, Codex CLI, Qwen Code, Windsurf
   - **優先度**: クリティカル (コア差別化要因)

2. **憲法ガバナンス (9条項)**
   - **ユーザー価値**: 実装前に品質基準を自動的に実施
   - **機能**: 
     - 条項I: ライブラリファースト原則
     - 条項II: CLIインターフェイス義務
     - 条項III: テストファースト命令 (Red-Green-Blue)
     - 条項IV: EARS要件フォーマット
     - 条項V: トレーサビリティ義務 (要件 ↔ 設計 ↔ コード ↔ テスト)
     - 条項VI: プロジェクトメモリ (ステアリングシステム)
     - 条項VII: バイリンガルドキュメント (英語/日本語)
     - 条項VIII: 単一の真実の情報源
     - 条項IX: テストでの実サービス (モックなし)
   - **優先度**: クリティカル (品質実施)

3. **EARS要件フォーマット**
   - **ユーザー価値**: 曖昧さを排除、テスト可能性を確保
   - **機能**: 5つのEARSパターン (イベント駆動、状態駆動、望ましくない動作、オプション機能、普遍的)
   - **優先度**: クリティカル (SDDの基盤)

4. **プロジェクトメモリ (ステアリングシステム)**
   - **ユーザー価値**: AIエージェントがプロジェクトコンテキストを自動学習 (アーキテクチャ、技術スタック、ビジネスドメイン)
   - **機能**: \`structure.md\`, \`tech.md\`, \`product.md\` を自動維持
   - **優先度**: クリティカル (コンテキスト継続性)

5. **25のClaude Codeスキル** (Claude Code専用)
   - **ユーザー価値**: すべてのSDDステージ用の専門AIエージェント
   - **機能**: オーケストレーター、ステアリング、要件、アーキテクチャ、開発、品質、セキュリティ、インフラ、ドキュメント
   - **優先度**: 高 (Claude Codeユーザー)

6. **グリーンフィールドとブラウンフィールドワークフロー**
   - **ユーザー価値**: 新規プロジェクト (0→1) と既存コードベース (1→n) の両方をサポート
   - **機能**:
     - グリーンフィールド: 8ステージSDDワークフロー (調査 → モニタリング)
     - ブラウンフィールド: デルタ仕様 (ADDED/MODIFIED/REMOVED)、変更影響分析
   - **優先度**: クリティカル (実世界の柔軟性)

7. **完全なトレーサビリティ**
   - **ユーザー価値**: 設計、コード、テストを通じてすべての要件を追跡
   - **機能**: トレーサビリティマトリックス、監査証跡、憲法検証
   - **優先度**: 高 (コンプライアンス、保守)

8. **バイリンガルドキュメント**
   - **ユーザー価値**: すべての生成ドキュメントが英語+日本語
   - **機能**: \`.md\` (英語プライマリ), \`.ja.md\` (日本語翻訳)
   - **優先度**: 中 (日本市場フォーカス)

### あると良い機能 (将来のロードマップ)

- **Visual Studio Code拡張機能**: musubiコマンド用GUI (将来)
- **GitHub Actions統合**: PRで憲法検証を自動実行 (将来)

- **メトリクスダッシュボード**: SDDワークフローメトリクスを経時的に追跡 (将来)

### 明示的にスコープ外

**MUSUBIはCLIツールであり、以下ではありません**:
- ❌ **ホストサービスではない** (クラウドインフラなし、SaaSなし)
- ❌ **AIモデルではない** (既存のAIエージェントを使用、モデルを訓練しない)
- ❌ **コードジェネレーターではない** (AIエージェント用テンプレート、コード生成自体ではない)
- ❌ **プロジェクト管理ツールではない** (課題追跡なし、カンバンボードなし)
- ❌ **CI/CDパイプラインではない** (テンプレート提供、パイプライン実行はしない)
- ❌ **テストフレームワークではない** (テストファースト実施、jest/vitest等を使用)
- ❌ **ドキュメントプラットフォームではない** (markdown生成、ホストドキュメントではない)

## ビジネスモデル

### 収益モデル
**オープンソース / 永久無料** - MUSUBIは自由に配布される公開npmパッケージです。

**収益化計画なし** - リファレンス実装と研究プロジェクトとして構築:
- SDDベストプラクティスを実証
- 6つの主要SDDフレームワークを統合
- 仕様駆動開発について開発者を教育
- マルチエージェントアーキテクチャパターンを展示

**潜在的な将来の収益源** (未実装):
- プレミアムテンプレート / スキルマーケットプレイス (コミュニティ駆動)
- エンタープライズコンサルティング (SDD採用、カスタム憲法条項)
- ホストSDDプラットフォーム (需要がある場合)

### 主要メトリクス
MUSUBIの成功指標 (コミュニティ/採用フォーカス):

**採用メトリクス**:
- 週あたりのnpmダウンロード (npm install, npx使用)
- GitHubスターとフォーク
- MUSUBIを使用しているプロジェクト数 (テレメトリーオプトイン経由、将来)
- コミュニティ貢献 (PR、イシュー、スキルテンプレート)

**品質メトリクス**:
- 憲法準拠率 (検証に合格したプロジェクトの%)
- MUSUBI生成コードのテストカバレッジ (目標: 80%+)
- トレーサビリティ完全性 (要件 → テストマッピング)

**コミュニティメトリクス**:
- ドキュメントページビュー (GitHub、Qiita記事)
- エージェント多様性 (AIエージェントタイプごとのユーザー%)
- スキル採用 (25スキルのどれが最も使用されているか)

### ペルソナ1: Kenji (Claude Codeパワーユーザー)
**背景**: 東京のスタートアップのシニアフルスタック開発者、8年の経験、毎日Claude Codeを使用
**目標**:
- 品質を維持しながらAIで機能をより速く構築
- チーム全体でアーキテクチャ標準を実施
- 要件からデプロイまでのトレーサビリティを維持

**ペインポイント**:
- Claude Codeが要件ドキュメントなしでコードを生成
- AI生成コードのレビューが困難 (この設計はどこから来た?)
- ジュニア開発者がAIコパイロット使用時にテストをスキップ
- AIで「ライブラリファースト」原則を実施する方法がない

**MUSUBIの使い方**:
- **毎日**: \`/sdd-requirements [feature]\` → \`/sdd-design [feature]\` → \`/sdd-implement [feature]\`
- **毎週**: \`/sdd-validate\` で憲法準拠をチェック
- **毎月**: \`@steering\` でアーキテクチャ変更後にプロジェクトメモリを更新
- **使用**: すべての25 Claude Codeスキル、特に複雑なワークフロー用の \`@orchestrator\`

### ペルソナ2: Sarah (GitHub Copilotユーザー、テックリード)
**背景**: 米国SaaS企業のテックリード、5人チーム管理、AIコーディングツールを評価中
**目標**:
- コード品質を犠牲にせずにAIアシスタントを採用
- チームがSOLID原則に従うことを確保
- コンプライアンス用の監査証跡を維持 (SOC 2)

**ペインポイント**:
- GitHub Copilotは速いが一貫性のないコードを生成
- Copilotで「テストファースト」を実施する方法がない
- チームが要件ドキュメントをスキップ ("Copilotは何をすべきか知っている")
- どのCopilot提案がどの要件を満たしたか追跡できない

**MUSUBIの使い方**:
- **毎日**: \`#sdd-requirements\` → \`#sdd-implement\` をGitHub Copilotチャット経由
- **毎週**: \`musubi validate\` CLIで憲法準拠をチェック
- **毎月**: トレーサビリティマトリックスをレビュー (要件 ↔ コード ↔ テスト)
- **使用**: コマンドのみ (GitHub CopilotはSkills APIをサポートしない)

### ペルソナ3: Taro (ブラウンフィールド移行エンジニア)
**背景**: レガシーモノリスをマイクロサービスにリファクタリングするバックエンドエンジニア、10年の経験
**目標**:
- 本番環境を壊さずにコードベースを段階的に近代化
- 影響分析ですべての変更を文書化
- AIを使用してリファクタリングを加速 (Cursor IDE)

**ペインポイント**:
- 変更されるコードに何が依存しているか把握困難
- AIが副作用を考慮せずに変更を提案
- "なぜこれが変更されたか"の監査証跡がない
- 不完全なリファクタリングによるリグレッションバグ

**MUSUBIの使い方**:
- **毎日**: \`/sdd-change-init [change]\` → 影響分析 → \`/sdd-change-apply\`
- **毎週**: \`/sdd-change-archive\` で完了した変更を文書化
- **毎月**: デルタ仕様をレビュー (ADDED/MODIFIED/REMOVED)
- **使用**: change-impact-analyzerスキルとブラウンフィールドワークフロー

## 製品原則

### 設計原則 (憲法ガバナンス)
1. **ライブラリファースト**: すべての機能は独立したライブラリとして開始 (条項I)
2. **CLIインターフェイス義務**: すべてのライブラリがCLIインターフェイスを公開 (条項II)
3. **テストファースト命令**: Red-Green-Blueサイクル実施 (条項III)
4. **EARS要件**: 明確な要件のみ (条項IV)
5. **100%トレーサビリティ**: 要件 ↔ 設計 ↔ コード ↔ テスト (条項V)
6. **プロジェクトメモリ**: 自動更新ステアリングコンテキスト (条項VI)
7. **バイリンガルファースト**: 英語+日本語ドキュメント (条項VII)
8. **単一の真実の情報源**: 矛盾するドキュメントなし (条項VIII)
9. **テストでの実サービス**: 統合テストにモックなし (条項IX)

### 開発優先順位 (MUSUBI自体)
トレードオフ時:
1. **シンプルさ > 機能**: CLIを使いやすく保つ、複雑さを避ける
2. **互換性 > 最先端**: Node.js 18+ (LTS)をサポート、最先端ではない
3. **ドキュメント > コード**: 包括的な例とガイド
4. **コミュニティ > 収益**: オープンソース、永久無料、コミュニティ駆動
5. **マルチエージェントサポート > 単一エージェント最適化**: すべての7エージェントで動作

## ドメイン用語

### 主要用語 (MUSUBI語彙)
一貫して使用する標準用語:

- **MUSUBI**: 究極のSDDツール ("結び" = 仕様、設計、コードを結ぶ/繋ぐ)
- **SDD**: 仕様駆動開発 (要件ファースト方法論)
- **EARS**: Easy Approach to Requirements Syntax (明確な要件のための5パターン)
- **憲法条項**: 9つの不変ガバナンスルール (条項I-IX)
- **ステアリングシステム**: プロジェクトメモリ (structure.md, tech.md, product.md)
- **スキル**: Claude Code専門エージェント (合計25、Skills API専用)
- **コマンド**: スラッシュコマンドまたはプロンプト (すべての7エージェントで動作)
- **エージェント**: AIコーディングアシスタントプラットフォーム (Claude Code, GitHub Copilot, Cursor等)
- **テンプレート**: \`src/templates/\` のソースファイル、ユーザープロジェクトにコピー
- **グリーンフィールド**: 新規プロジェクトワークフロー (0→1、完全な8ステージSDD)
- **ブラウンフィールド**: 既存コードベースワークフロー (1→n、デルタ仕様、変更影響)
- **デルタ仕様**: 変更仕様 (ADDED/MODIFIED/REMOVED要件)
- **トレーサビリティマトリックス**: 要件 ↔ 設計 ↔ コード ↔ テストのマッピング
- **Phase -1**: 実装前憲法検証ゲート

### 避けるべき用語 (曖昧または混乱を招く)
- ❌ MUSUBIスキルに"エージェント"と言わない ("スキル"または"専門スキル"と言う)
- ❌ Skills APIに"プロンプト"と言わない ("スキル"と言う、スキルはプロンプトより豊富)
- ❌ EARS要件に"テンプレート"と言わない ("EARSパターン"または"要件フォーマット"と言う)
- ❌ "ガバナンス"単独で言わない ("憲法ガバナンス" - 9条項を指定)
- ❌ "プロジェクトコンテキスト"と言わない ("プロジェクトメモリ"または"ステアリングコンテキスト"と言う)
- ❌ "仕様"単独で言わない ("要件仕様"、"設計仕様"、または"デルタ仕様"を指定)

## ユーザージャーニー

### 典型的なユーザーフロー (Claude Code、グリーンフィールドプロジェクト)

1. **インストール** (1回限りのセットアップ):
   \`\`\`bash
   npx musubi-sdd init --claude
   # 選択: グリーンフィールド、すべての25スキル
   \`\`\`
   → \`.claude/skills/\`, \`.claude/commands/\`, \`steering/\`, \`templates/\`, \`storage/\` をインストール

2. **プロジェクト初期化**:
   - \`/sdd-steering\` → コードベース分析、\`structure.md\`, \`tech.md\`, \`product.md\` 生成
   - ステアリングファイルをレビュー、必要に応じて編集
   - 憲法条項が有効化 (9条項を実施)

3. **機能開発** (毎日):
   - \`/sdd-requirements [feature]\` → EARS要件を作成
   - \`/sdd-design [feature]\` → C4図、ADR、API仕様を生成
   - \`/sdd-tasks [feature]\` → 実装タスクに分解

   - Markdownフォーマット

   - \`.github/prompts/\` に9プロンプト
   - Markdownフォーマット、\`#\` プレフィックス

3. **Cursor IDE** (コマンドのみ)
   - \`.cursor/commands/\` に9コマンド
   - Markdownフォーマット

4. **Gemini CLI** (コマンドのみ)
   - \`.gemini/commands/\` に9コマンド
   - **TOMLフォーマット** (Gemini固有)

5. **Codex CLI** (コマンドのみ)
   - \`.codex/prompts/\` に9プロンプト
   - Markdownフォーマット、\`/prompts:\` プレフィックス

6. **Qwen Code** (コマンドのみ)

   - Markdownフォーマット

## コンプライアンス & 規制

### 要件
**適用外** - MUSUBIはローカルCLIツール、ホストサービスではありません。

**データ収集なし**:
- ユーザー追跡や分析なし
- テレメトリーなし (将来のオプトイン検討)
- クラウドストレージなし (すべてのファイルはローカル)
- 認証やユーザーアカウントなし

**ライセンス**: MIT (オープンソース、寛容)

**プライバシー**: ユーザーのプロジェクトは自分のマシンに残り、MUSUBIは決して見ません。
