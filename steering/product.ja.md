## 概要

MUSUBI (「むすひ」- 繋ぐ/結ぶ に由来) は、6つの主要フレームワークのベストフィーチャーを統一的で包括的なプラットフォームに統合した**次世代仕様駆動開発 (SDD) ツール**です。ソフトウェア開発をコードファーストから仕様ファーストに変革し、開発ライフサイクル全体にわたってトレーサビリティ、品質、ガバナンスを確保します。

### ビジョンステートメント

**「仕様駆動開発を世界中のソフトウェアチームの新しい標準にする」**

- ソフトウェアチームはコードを書く前に明確でテスト可能な仕様から始める
- 要件、設計、コード、テストが完全に追跡可能で同期している
- 品質は手動レビューではなく憲法ガバナンスを通じて強制される

6. **コミュニティ**: 協力的な開発、透明なロードマップ

## 製品目的

### MUSUBIとは?

MUSUBIは次のように実装された**包括的なSDDフレームワーク**です:
- **25のClaude Code Skills**: 全8 SDDワークフローステージをカバーする専門AIエージェント
- **憲法ガバナンスシステム**: 9つの不変条項 + Phase -1 Gates
- **デルタ仕様システム**: ブラウンフィールドプロジェクト用の変更管理
- **自動更新プロジェクトメモリ**: アーキテクチャ、技術スタック、ビジネスコンテキストを追跡するステアリングシステム
- **マルチエージェントオーケストレーション**: インテリジェントなスキル選択と並列実行
- **完全なトレーサビリティ**: 要件 ↔ 設計 ↔ コード ↔ テストのマッピング

### 問題ステートメント

**現状**: ソフトウェアチームは、断片化されたツールのために仕様駆動開発に苦労しています。

**MUSUBIが解決する問題**:

1. **断片化**: 既存のSDDフレームワークはそれぞれ異なる領域で優れているが、包括的なカバレッジに欠ける
   - musuhi: 20エージェント、しかしブラウンフィールドサポートなし
   - OpenSpec: デルタ仕様、しかしエージェントシステムなし
   - ag2: マルチエージェントオーケストレーション、しかしSDD固有ではない
   - ai-dev-tasks: シンプルなオンボーディング、しかし限定的 (2ファイルのみ)
   - cc-sdd: 良好なワークフロー、しかしKiro依存
   - spec-kit: 憲法ガバナンス、しかし硬直的

2. **不完全なワークフロー**: 全8 SDDステージ (調査 → モニタリング) をカバーする単一ツールが存在しない

3. **不十分なガバナンス**: 品質ゲートと憲法原則の手動強制

4. **限定的なブラウンフィールドサポート**: ほとんどのツールはグリーンフィールド (0→1) プロジェクトのみに焦点を当てている

5. **トレーサビリティ不足**: 要件 ↔ コード ↔ テストのマッピングは手動でエラーが発生しやすい

6. **ベンダーロックイン**: 特定のIDEまたはAIプラットフォームに縛られたツール

**これらの問題の影響**:
- 仕様とコードの不整合による20-30%のやり直し率
- 機能ごとに3日以上かかる手動仕様作成
- チーム間での一貫性のない品質とガバナンス
- 既存コードベースの適応の困難さ (ブラウンフィールドプロジェクト)

### ソリューション: MUSUBIのアプローチ

**ベストプラクティスの統合**:
- **musuhiから**: 25の専門スキル、自動更新ステアリング
- **OpenSpecから**: 変更管理用のデルタ仕様
- **ag2から**: マルチエージェントオーケストレーションパターン (自動、シーケンシャル、並列、ネスト、Human-in-the-Loop)
- **ai-dev-tasksから**: 段階的複雑性 (初心者向けシンプルな2ファイルPRD)
- **cc-sddから**: 8ステージSDDワークフロー (調査 → モニタリング)
- **spec-kitから**: 憲法ガバナンス (9条項 + Phase -1 Gates)

**独自の差別化要因**:
1. **完全カバレッジ**: 8 SDDワークフローステージの100%
2. **自動ガバナンス**: constitution-enforcerスキルがすべてのアーティファクトを検証
3. **デュアルプロジェクトサポート**: グリーンフィールド (0→1) とブラウンフィールド (1→n)
4. **プラットフォーム非依存**: 13以上のAIコーディングアシスタントと連携
5. **段階的複雑性**: 2ファイルPRDから25スキルオーケストレーションまでスケール
6. **ゼロベンダーロックイン**: Markdownベース、移植可能、オープンソース

## コア機能

### 1. 完全なSDDワークフロー (8ステージ)

**ステージ1: 調査**
- **スキル**: steering, performance-optimizer
- **成果物**: プロジェクトメモリ (ステアリングファイル)、技術調査

**ステージ2: 要件**
- **スキル**: requirements-analyst, project-manager, change-impact-analyzer
- **成果物**: EARS形式要件、プロジェクト計画、変更提案

**ステージ3: 設計**
- **スキル**: system-architect, api-designer, database-schema-designer, ui-ux-designer
- **成果物**: C4図、ADR、API仕様、DBスキーマ、UIワイヤーフレーム

**ステージ4: タスク**
- **スキル**: orchestrator, project-manager
- **成果物**: P-ラベル付きタスク (並列実行計画)

**ステージ5: 実装**
- **スキル**: software-developer, code-reviewer, bug-hunter
- **成果物**: コード、バグ修正、リファクタリング

**ステージ6: テスト**
- **スキル**: test-engineer, quality-assurance, security-auditor
- **成果物**: ユニット/統合/E2Eテスト、QA計画、セキュリティ監査

**ステージ7: デプロイメント**
- **スキル**: devops-engineer, cloud-architect, database-administrator, release-coordinator
- **成果物**: CI/CDパイプライン、インフラコード、リリース計画

**ステージ8: モニタリング**
- **スキル**: site-reliability-engineer
- **成果物**: SLO/SLI定義、モニタリングダッシュボード、インシデントランブック

### 2. 憲法ガバナンス

**9つの憲法条項** (不変、constitution-enforcerスキルによって強制):
1. **ライブラリファーストの原則**: すべての機能はライブラリとして始まる
2. **CLIインターフェース義務**: すべてのライブラリはCLIを公開する必要がある
3. **テストファーストの命令**: テストなしにコードはない (Red→Green→Blue)
4. **EARS要件フォーマット**: すべての要件はEARSパターンを使用
5. **トレーサビリティ義務**: 100%の要件 ↔ 設計 ↔ コード ↔ テスト
6. **プロジェクトメモリ**: スキルは作業前にステアリングをチェック
7. **シンプリシティゲート**: ≤3プロジェクト、将来の備えなし
8. **抽象化禁止ゲート**: フレームワークを直接使用、ラッパーなし
9. **統合ファーストテスティング**: 実サービス、契約必須

**Phase -1 Gates** (実装前検証):
- すべてのゲートは実装開始前に合格する必要がある
- constitution-enforcerスキルによる自動検証
- ソースで技術的負債を防止

### 3. デルタ仕様システム

**グリーンフィールドプロジェクト (0→1)**:
- `storage/specs/[capability]/`に完全な仕様
- ゼロからの完全な要件、設計、タスク

**ブラウンフィールドプロジェクト (1→n)**:
- `storage/changes/[change-id]/`に変更提案
- change-impact-analyzerスキルによる影響分析
- デルタファイルを介して元の仕様へのトレーサビリティ
- 完了した変更をアーカイブ

**ワークフロー**:
```
/sdd-change-init → 提案 → 影響分析 → 実装 → アーカイブ
```

### 4. 自動更新プロジェクトメモリ

**ステアリングファイル**:
- `steering/structure.md`: アーキテクチャパターン、ディレクトリ構成
- `steering/tech.md`: 技術スタック、フレームワーク、ツール
- `steering/product.md`: ビジネスコンテキスト、製品目的、ユーザー (このファイル)

**自動更新トリガー**:
| 完了したスキル | 更新内容 |
|-----------------|---------|
| system-architect | structure.md (アーキテクチャパターン) |
| api-designer | tech.md (API規約) |
| database-schema-designer | tech.md (DBパターン) |
| cloud-architect | tech.md + structure.md (インフラ) |
| software-developer | tech.md (新しい依存関係) |
| requirements-analyst | product.md (新機能、ビジネスルール) |

**利点**:
- 常に最新のプロジェクトコンテキスト
- 陳腐化したドキュメントなし
- すべてのスキルが現在の真実を参照

### 5. マルチエージェントオーケストレーション

**5つのオーケストレーションパターン** (ag2から):
1. **自動選択**: オーケストレーターが意図を分析 → スキルを選択 → 実行
2. **シーケンシャル**: 線形ワークフロー (調査 → 要件 → 設計 → ...)
3. **並列**: P-ラベル付きタスクを同時実行
4. **ネストされた委譲**: 複雑なドメイン用のサブオーケストレーター
5. **Human-in-the-Loop**: ユーザー承認が必要な検証ゲート

**例**:
```
ユーザー: "ユーザー認証機能を構築"
  ↓
orchestrator (自動選択):
  1. requirements-analyst → EARS要件
  2. system-architect + api-designer + database-schema-designer (並列)
  3. software-developer → 実装
  4. test-engineer → テストスイート
  5. constitution-enforcer → コンプライアンス検証
```

### 6. 完全なトレーサビリティ

**トレーサビリティマトリックス**:
```
要件 → 設計 → コード → テスト
[REQ-001]   → [ARCH-001] → [src/auth.ts] → [test/auth.test.ts]
```

**強制者**:
- `traceability-auditor`スキル
- `validators/traceability.py`スクリプト
- 憲法条項V (トレーサビリティ義務)

**利点**:
- 100%要件カバレッジを証明
- 孤立コード (要件のないコード) を検出
- 不足しているテスト (テストのない要件) を検出
- コンプライアンス用の監査証跡 (SOC2, ISO27001)

4. 変更設計: `api-designer`がAPI仕様を更新 (後方互換性)
5. 段階的実装: `software-developer` + `test-engineer` (フィーチャーフラグ)
6. トレーサビリティ検証: `traceability-auditor`が100%カバレッジを確認
7. 段階的デプロイ: `release-coordinator`がロールアウトを管理 (10% → 50% → 100%)
8. 変更アーカイブ: `/sdd-change-archive` → アーカイブに移動、仕様を更新

**結果**: 6週間で移行完了 (MUSUBIなしでは12週間)、ダウンタイムゼロ、コンプライアンス用の完全監査証跡

### ジャーニー3: オープンソース貢献 (新規貢献者)

**ユーザー**: Sam、MUSUBIプロジェクトへの初めての貢献者

7. 検証合格: すべてのPhase -1 Gatesが合格 → PR承認
